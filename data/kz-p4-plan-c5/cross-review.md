## 2. Where the two drafts already agree

Both plans independently converged on: SpacetimeDB Cloud only, no self-hosting; both-track generated server registry emitted by the track compiler with a `track:check`-style drift gate; the handshake as a frozen `{compiler_version, gameplay_hash}` tuple on the match row checked at join, with a dedicated client `incompatible-version` UI state; multi-match schema replacing the `activeRace()` singleton; preserving the N0 lifecycle rules (ready/countdown/15 s reconnect grace/DNF/results lock/rematch); event tables for combat/VFX fan-out; match-scoped filtered subscriptions replacing `SELECT *`; ship-select as validated cosmetic state on the participant row; per-track checkpoint counts ending the `0..7` hardcode; generated (never hand-typed) shared constants; a deterministic simulated-latency harness plus bot soaks plus piloted QA; anti-cheat proportionality ("P4 does not need invasive client anti-cheat", x9:240 — same trust model as c5 §6); and the P3 exit gate being currently unmet. The captain should read the disagreement below knowing it sits on top of ~80 % structural agreement.

## 3. The netcode disagreement, on the merits

x9 recommends: server-authoritative 60 Hz sim advanced 3 substeps per 50 ms scheduled reducer; clients send only input batches; a new pure-TS movement core (`module/src/sim-core/`) replaces Rapier for online movement on both server and client-predictor; own craft predicted/reconciled; remotes interpolated 100–150 ms behind (x9:7-9, 126-138).

c5 recommends: clients keep Rapier-simulated authority over their own craft inside generated caps (extending N0's validated relay); the server owns all discrete outcomes (laps, pickups, inventory, damage, kills, respawns, results); combat resolves per weapon class (server-instant Arc/EMP, server-scheduled Rail, victim-triggered mines, shooter-claimed Pulse/Seeker under server plausibility).

### 3.1 Points x9 lands against c5 — conceded, with fixes adopted

1. **Movement cheat ceiling.** x9:363 is right that c5's model "cannot prove track contacts [or] checkpoint crossings" in the strong sense: a modified client could fly the chord of a hairpin at legal speeds. c5's caps bound speed and teleports but not *path legality*. **Adopted into c5:** artifact-based pose plausibility in `publish_transform` (published pose must lie within the track corridor envelope — the artifact ships `width_mm`, `up`, and a spatial index for exactly this query — or be in a bounded fall-recovery window), and `cross_checkpoint` additionally requires the caller's current transform row to be inside the claimed gate OBB (+staleness tolerance). This closes corner-cutting and remote-gate-claiming at 15 Hz granularity; the residual (perfect racing lines) is the aimbot-class residue both models concede (x9:250: "A modified client can automate valid inputs").
2. **Join-outcome ergonomics.** x9's `admission` row (x9:150) is better engineering than c5's structured-error-string parsing: subscribable, carries reason codes and expected/received tuples, testable. **Adopted** (with one caveat: RLS availability on the 2.6 TS SDK is itself a spike item; fallback is a public admission row carrying only reason codes, which leaks nothing sensitive).
3. **`balance_version` alongside the track tuple** (x9:192): correct observation that gameplayHash covers track data, not combat/energy constants. c5 had `protocol_version` on `server_info`; adding a balance version to the match row is strictly better. **Adopted.**
4. **Duplicate-tab / connection-generation policy** (x9:213, 378): a real hole in c5 — two tabs share one identity and both pass the N0 authority check (`module/src/index.ts:495`), so they would fight over one transform row. **Adopted** (newest authenticated connection supersedes).
5. **Private `room_secret` table** (x9:164) — c5 put the join code on the public match row, which is scrapable. **Adopted.**
6. **Operational shape**: dev/stage/prod Maincloud databases, canary client, never `--delete-data` on prod, N0 DB retired read-only (x9:277-287); a 45 s post-winner finish window instead of only the 300 s global timeout (x9:210, 379); budget-style acceptance numbers with published histograms (x9:322, 329). All **adopted** — x9's release engineering is more complete than c5's.
7. **Late-join/rejoin threat reconstruction**: x9's durable `combat_entity` rows mean a rejoining client reconstructs in-flight seekers/mines (x9:154, 227). c5 has rows for mines/rail only; pulse/seekers are client-visual and a rejoiner misses transient threats for a second or two. Minor, but x9 is more complete; c5 can adopt seeker rows cheaply since seekers live 4.5 s.

### 3.2 Where x9's model breaks against the actual platform and codebase

**(a) The transport turns x9's worst case into own-craft corrections — the one place this product cannot afford them.** SpacetimeDB gives exactly one channel: reliable, ordered WebSocket (x9 concedes at x9:130: "WebSocket delivery is reliable/ordered... no UDP-style redundant packet scheme"). x9's own input policy is: reject frames more than +2 ticks ahead (~33 ms of server-side buffer), hold a missing input sticky for ≤ 6 ticks, then force neutral (x9:130). And the server never rewinds ("No victim rewind and no retroactive projectile hit", x9:226; nothing anywhere re-simulates past ticks when late input arrives). Now walk a routine event through it: one lost TCP segment at 1 % loss and 100 ms RTT stalls *all* subsequent frames for a retransmit (≥ 1×RTT with fast-retransmit, frequently 200 ms+ with RTO floors). The server exhausts the 6-tick sticky window and simulates **neutral input** — thrust off, steer centered — for the remainder of the stall, while the client's predictor continued with the player's real inputs. When the stall clears, those frames are stale (older than the tick the server already simulated), so they are dropped, and reconciliation snaps/smooths the player's own craft to a trajectory they never drove. At 70 m/s into a corner, 200 ms of neutral-vs-steering divergence is far beyond x9's own 3 m/12° snap threshold (x9:133). Expected frequency at 1 % loss and 20 packets/s upstream: roughly one such event every ~5 s of racing. Under c5's model the identical burst delays remote-ghost updates and outcome events; **your own craft is untouched by construction**, because nothing upstream is allowed to correct it. For a product whose bar is the P1.2/P1.4 feel contract, this is the decisive asymmetry: x9 concentrates network pathology on the local craft; c5 concentrates it on remote presentation. x9's harness thresholds ("under 150 ms RTT/20 ms jitter local correction p95 <1 m", x9:322) are specified only for clean-ish conditions and its plan contains no mitigation for TCP head-of-line bursts — no server-side input rollback, no larger input-delay buffer (capped at +2 ticks), no unreliable channel to switch to.

**(b) Racer-racer contact — x9's headline gameplay advantage — is also its guaranteed misprediction source.** x9 correctly sells symmetric authoritative contacts as something c5 v1 lacks. But the client predictor cannot know another racer's authoritative pose; it has 100–150 ms-old interpolation (x9:133). Every server-resolved shunt is therefore resolved against positions the local predictor could not have used: at 70–88 m/s closing speeds, up to ~10-13 m of unknowable relative displacement. The result is that contact — precisely the chaotic, high-adrenaline moment — is when the player's own craft gets yanked to the server's version of the collision. x9's §"Why 20 Hz replication" (x9:136-138) sweeps checkpoints and pickups but never addresses contact misprediction, and its correction budget (p95 < 1 m) cannot survive pack contact by simple kinematics. So the honest comparison is not "x9 has contacts, c5 doesn't"; it is "x9 has authoritative contacts *that mispredict on your own hull*, c5 v1 has non-colliding ghosts (N0 status quo) with an optional local-kinematic flag (c5 M7)". Which is preferable is a product call, but x9's draft presents only the upside.

**(c) The pure movement core is a phase-sized research project with fail-stop semantics, sitting at the head of the plan.** x9 prices the port as one PR (x9:344: sim-core `input,track,craft,contact,state,hash` + browser adapter + benchmark + A/B gate) and simultaneously declares it the highest risk with a stop-the-line rule: "If it cannot pass without constant correction, stop before schema and either improve the core or prove Rapier-in-module feasibility; **do not ship client-transform authority**" (x9:374). Three problems:
   1. *Scale.* The port must reproduce: 4-pad suspension against a curved, banked (45-50° on Foundry) ribbon; slab containment at up to 105 m/s; wall glances with the tuned restitution/friction-combine behavior; airborne/landing; racer contact impulses; and the whole thing bit-stable and feel-identical. The repo's own history says surface containment at speed is hard *even with a real physics engine* — the fall-through saga is documented as a sharp edge (`CLAUDE.md`: thin trimesh tunnels at speed; slab extrusion + hull filter fix; a dedicated real-Rapier regression gate `src/game/physics/fallthrough.test.ts` in `pnpm test`). x9 re-opens that bug class in fresh code, server-side, where a containment failure is an authority failure for all eight players.
   2. *Plan shape.* Schema freeze (x9 PR 2) explicitly waits on the spike ("its required state fields define the frozen P4 protocol", x9:16), so rooms, admission, and **the captain-mandated handshake** all queue behind the least predictable work item. If PR 1 slips or fails, P4 delivers nothing while it is "improved" — and the stated alternative escape hatch (Rapier-in-module) is one x9's own rejected-alternatives table calls unproven (x9:365).
   3. *Permanent fork.* Online ships a second handling model (pure core + kinematic proxies) while solo keeps dynamic Rapier (x9:131 "Solo continues using dynamic Rapier"). Every future tuning change pays a parity-revalidation tax across two integrators forever, and solo↔online skill transfer degrades in exactly the subtle ways an A/B trace gate is worst at catching. This contradicts the anti-fork principle x9 itself applies to inputs ("do not invent a second input vocabulary", x9:25).

**(d) The no-rewind combat policy breaks the roster's bread-and-butter weapon.** x9 grants bounded lag compensation to Arc only ("Only instant weapon granted bounded lag compensation... cap Arc use at six ticks initially", x9:229, 234). Pulse — the most common offense drop — spawns server-side at the accepted tick and swept-tests against **current** server poses with "No victim rewind and no retroactive projectile hit" (x9:226). But the shooter aimed at a remote craft rendered 100–150 ms in the past (x9:133). At 130 m/s bolt speed against 70–88 m/s targets with a 1.6 m hit radius (`weaponTuning.ts:73-94`), a crossing or fleeing target displaces multiple hit-radii within the staleness window: visually-perfect shots systematically miss, and the landed hit-confirm feel (`notifyHitConfirmed`, hitstop) arrives only after the server event round-trip (x9:226: "Hit marker/damage only after server event"). This is the exact failure mode c5's §2.2-Option-B analysis predicted for server-side hit-testing on stale poses, and it is why c5 resolves travel-time projectiles in the shooter's perceived frame under server plausibility caps. The fix exists *inside x9's own architecture* — extend the 12-tick pose history rewind from Arc to projectile sweeps — but x9 withholds it precisely where it's most needed, without justification. If x9 is chosen, this must change.

**(e) Server load and real-time coupling are strictly worse, on x9's own numbers, and unmeasured.** Per active match: 20 `match_tick` tx/s (each simulating 24 racer-substeps plus entities and writing 8 public + 8 private rows + history pruning) **plus** 8 × 20 Hz input batches = 160 tx/s inserting ~480 `input_frame` rows/s that must then be deleted (x9:129-132, 166). c5's steady state is 120 small validate-and-write transactions/s and zero sustained sim CPU. Neither plan has Maincloud measurements (both say so), but the asymmetry matters: c5's unknowns are billing-shaped (cost per small tx), while x9's unknowns are *latency-shaped* — a billed, shared, transactionally-serialized system must now hit a soft-real-time p95 (x9:329 budgets scheduled callbacks < 25 ms) with input transactions from every player and every concurrent match contending against every match's tick transaction. x9's own overrun policy — kill the race after 12 ticks of debt for 3 callbacks ("end the match with `server_overrun`", x9:129) — is a failure mode with no analog in c5, and terminating a live 8-player race because a shared scheduler fell behind is the harshest possible resolution of a risk the plan chose to take on.

**(f) For everything a player sees about *other* racers, the models are equivalent.** x9 renders remotes from 20 Hz snapshots interpolated 100–150 ms behind (x9:133); c5 renders remotes from 15 Hz rows interpolated 120 ms behind (`transformSync.ts:162`). Remote fidelity, telegraph lead times, and threat readability are a wash. x9's user-visible deltas reduce to exactly two: authoritative contacts (with the §3.2(b) caveat) and the anti-cheat ceiling. Its costs — §3.2(a) own-craft corrections, (c) the movement-core program and permanent fork, (d) degraded projectile feel, (e) real-time server coupling — all land on things this product has explicitly prioritized. That is the core of the disagreement.

### 3.3 x9 arguments that do not actually discriminate between the models

1. **"The current 80 m/s validator already rejects legitimate 88 m/s boost... illustrating why input authority is safer than maintaining heuristics"** (x9:246, 363). The drift is an argument for *generating* the constants — which both plans do — not for abandoning pose validation. x9's sim-core freezes the same tuning constants server-side and rots identically without codegen. Post-codegen, this argument is void.
2. **"Trust client hit/pickup reports... A browser can fabricate target, timing, socket, aim, damage, and countermeasure order"** (x9:364) — misstates c5's actual design: pickups are server-arbitrated with position validation (not client reports); damage is server-table-only (claims carry no damage field); countermeasure ordering (Specter/shield/EMP windows) resolves on the server clock; inventory is server-owned so you cannot fire unheld weapons. The genuine residual is fabricated *hit geometry inside plausibility envelopes* for Pulse/Seeker — i.e., a cheater who always "lands" legal shots he was allowed to fire. That is aimbot-equivalent, which x9 admits its own model cannot prevent either (x9:250). Note also there is no occlusion to fabricate through: the landed combat world performs pure distance/cone tests with no track occlusion even offline (`combatWorld.ts:1210-1233`, `:673-727`), so "shooting through walls" is not a delta.
3. **"High-speed weapons need a single durable adjudicator"** (x9:360) — c5 has one: every outcome (damage, kill, refund, respawn, pickup) settles in reducers against server state. The models differ in who measures *geometry* for two weapon families, not in who adjudicates.

## 4. What the converged plan should take from each draft

**Netcode recommendation for the converged plan: keep c5's model** — client movement authority under generated caps + server-arbitrated discrete outcomes + per-weapon combat split — hardened with x9's validation ideas. The deciding merits: the platform's reliable-ordered transport makes x9's failure modes land on the player's own craft (§3.2(a)(b)) in a product whose bar is feel; x9's plan head is an unbounded parity program with fail-stop semantics that queues the mandated handshake behind it (§3.2(c)); and x9's combat policy regresses the most-used weapon (§3.2(d)). x9's genuine advantages (movement cheat ceiling, symmetric contacts) are worth less than their price *for this product tier* — the brief itself asks for anti-cheat "proportionate to a browser game". If the captain instead weighs cheat-proof movement and authoritative contacts above own-craft stability and ~2-3× scope, x9 is the shape to build — but then with three mandatory modifications: extend pose-history rewind to projectile sweeps, replace the `server_overrun` match kill with degradation, and decouple the 0A/0B balance program from the schema-freeze critical path.

**Adopt from x9 into the converged plan (regardless of model):**
1. `admission` row for join outcomes incl. expected/received tuples (drop c5's error-string parsing); reason codes for incompatible/full/closed.
2. `room_secret` private-code table; `matchmaking_ticket` if quick-match needs fairness beyond "oldest open lobby".
3. `balance_version` + `protocol_version` on the match row, distinct reason codes from track-hash mismatch.
4. Connection-generation / duplicate-tab supersession policy, fixed before freeze.
5. Artifact-corridor pose plausibility + gate-OBB proximity checks on the relay (x9's critique, c5's mechanism — §3.1.1).
6. Maincloud dev/stage/prod databases, canary rollout, no in-place destructive migration of the N0 DB, N0 retired read-only.
7. 45 s post-winner finish window (piloted) replacing sole reliance on the 300 s timeout.
8. Budgeted, histogram-published acceptance criteria in the latency harness (correction/telegraph/bandwidth percentiles), and x9's fuller QA matrices (both-track × 2-and-8 players × throttle tiers; mismatch matrix incl. wrong-compiler-right-hash and reconnect-across-hot-swap).
9. Durable `combat_entity`-style rows extended to seekers so rejoiners reconstruct threats.
10. The sharper evidence items c5's draft lacked: quantize hash coverage of respawnPoses/spatial, `Scene.tsx` Neon-gated AiField, Foundry's checked-in tuple `2/3445598386`.

**Keep from c5 (x9 lacks or under-specifies):**
1. The whole shipping-increment shape: rooms + handshake land in the first two PRs on the proven N0 relay, independent of any physics research; combat arrives system-by-system with per-weapon resolution classes.
2. Per-weapon latency reasoning tied to landed telegraph durations (rail 0.65 s, mine arm 0.45 s ≫ RTT) and victim-triggered mines (the victim's own pose is already the authoritative quantity — exact, latency-immune, and un-spoofable beyond existing caps).
3. Lazy energy settlement folded into `publish_transform` + edge-scheduled zero-crossing — no sustained server tick loop at all.
4. Shooter-frame resolution for Pulse/Seeker preserving offline hit feel (with x9-style caps).
5. Local combat world reuse via a *generated shared pure-rules package* — the convergence of c5's `combatTuning.generated.ts` and x9's sim-core idea, scoped to combat geometry/energy/rolls and explicitly **excluding movement**; both drafts independently proposed the same generated-package fallback for the build boundary (x9:376, c5 §3.1), so make it the primary plan, not the fallback.

**New synthesis neither draft had: split the schema freeze.** Lifecycle/rooms/movement-relay schema has zero balance dependency — freeze it at the first module PR so the handshake ships immediately. Combat-constants freeze (`balance_version` 1) happens later, gated on whichever P3 balance evidence the captain selects: (a) c5-M0 racing-bands-both-tracks (cheap, no AI item use), or (b) x9-0A/0B's full item-using statistical program (expensive, runs in parallel, gates only the combat milestones). This removes the single worst schedule coupling in x9 and the single vaguest gate in c5.

## 5. Bottom line

x9 is a rigorous, honest draft whose repo evidence all checks out and whose schema/operational engineering is in several places better than mine — the converged plan should take those pieces wholesale. Its architecture, however, spends the project's scarcest resources (feel stability under real browser networks, schedule predictability, maintenance of one handling truth) to buy properties this product tier did not ask for, and its own risk register acknowledges the bet ("highest risk... stop before schema", x9:374) while its milestone plan prices that bet at one PR. The strongest single sentence against it is built from its own admissions: a reliable-ordered transport (x9:130), a ≤33 ms server input buffer (x9:130), neutral-fill after 100 ms (x9:130), no rewind (x9:226), and reconciliation snapping the local craft (x9:133) — chained together, that is "network jitter steers your ship", and no later milestone can tune it away because it is the architecture. The relay model's equivalent worst case — a remote ghost stutters and a hit confirms late — is the kind of degradation a browser racer can live with.

Recommended captain decision: converge on c5's netcode model with the §4 adoption list, split the schema freeze as above, and put the P3 balance-program choice (cheap racing gate now vs. full item-AI program in parallel) to the captain explicitly.
