## 2. Concrete disagreements, with rationale and a recommended resolution

**D1 — URL strategy: S8's `/workspace/[serviceLine]/**` vs F3's same-URL cookie-proxy rewrite.**
S8 is right that a route group alone can't fork the same URL (my design needed a proxy for exactly that reason), and their `[serviceLine]` path segment is genuinely valuable (deep links, audit clarity, line-scoped saved views). My residual objections to their framing: (a) if `/workspace/**` is treated as a *temporary pilot namespace*, cutover ends with either a mass rename or permanently migration-scented URLs; (b) two URL spaces during pilot means notifications/bookmarks minted in one space break in the other.
**Resolution: adopt S8's URL space and declare it the permanent product IA now** — `/workspace/[serviceLine]/…` is the final shape, not a pilot artefact; all new-surface links (notifications, ⌘K, emails later) mint only workspace URLs from day one; legacy routes 301 into it at cutover. This keeps S8's simplicity and kills the rename debt. Drop my proxy entirely (one less moving part in the auth-critical request path).

**D2 — Referral detail: S8's three-pane evidence workspace as the centre vs F3's tabbed workspace (Overview default).**
S8's evidence-reconciliation pane model is the right *implementation of evidence review* and better than my Evidence-tab spec; I concede the interior. But as the default landing for every referral visit it front-loads the heaviest surface (PDF viewer + three panes) on every persona and every revisit, including high-confidence complete referrals and roles (finance, area manager) whose job isn't field verification. At 1280–1440px, 38% source + 37% record + 25% rail ≈ a 480–540px source pane — uncomfortably close to the centre-column squeeze both audits criticised in the current UI. S8 itself concedes tablet degrades to tabs.
**Resolution: merge — tabbed workspace shell (F3) whose Evidence tab is S8's three-pane workspace, with a state-aware default tab:** referrals arriving with `needs_review`, low evidence coverage, or contradictions open directly on Evidence; clean high-band referrals open on Overview with the readiness banner one click from Evidence. This preserves S8's "evidence before inference" where it earns its cost and honours proportional oversight everywhere else. Pane proportions persist per user (S8's detail, keep).

**D3 — Typography: S8 keeps Source Sans 3 (operational) + demoted Source Serif 4 (reports) + Plex Mono vs F3's Inter + Plex Mono.**
I proposed Inter as a clean-break signal; S8 re-roles the existing OFL families. On reflection S8 is right: the captain's "unacceptable" verdict is about hierarchy, evidence and workflow, not the sans face; Source Sans 3 at operational sizes is unobjectionable; keeping loaded fonts is zero-cost; and *demoting* serif to long-form reports/evidence packs is better than my *deleting* it (inspection documents genuinely benefit).
**Resolution: concede to S8**, with two F3 riders: tabular-nums mandate on all data (shared), and record Inter as the alternative if the captain's brand review (S8's P0 brand decision) wants a fresh face — a one-token swap under either system.

**D4 — Brand colour: S8 teal `#0B5F5A` vs F3 blue `oklch(0.51 0.13 255)`.**
Genuinely subjective; both pass AA; teal signals clean break and avoids default-clinical-blue, blue keeps Registry Blue lineage and colour-culture familiarity. Neither draft can settle this — it's the captain's branding call (both drafts already list it).
**Resolution: put both swatches in the captain question; default to S8's teal if a visible clean break is wanted** (consistent with "rebuild, not reskin"), authored in OKLCH per F3's token format regardless. Adopt F3's triple structure (solid/tint/on-tint per semantic family — it's what fixes the current chip-contrast failures) and F3's confidence-band + risk-ramp tokens; adopt S8's rule that CQC domain accents are edge/icon accents never status colours.

**D5 — RBAC: S8's role + scope + capability model now vs F3's keep-5-roles + add finance.**
S8's end-state is right and my draft under-reached (a `finance` bolt-on doesn't express service-line scope, and multi-line makes scopes load-bearing). But S8 puts a policy-engine rewrite (row/service scopes, capability grants, scoped source-download permission) inside the same programme as a ground-up UI rebuild — that's the highest-blast-radius seam in the product (WS-7 explicitly frozen in both drafts!) churning while everything above it changes.
**Resolution: phase it.** Phases 0–2: keep the existing 5-role RBAC untouched; introduce S8's capability *names* (`clinical:decide`, `commercial:approve`, `evidence:verify`, `source:download`…) as a pure static derivation from the current roles behind the existing `can()` seam — code speaks capabilities from day one, policy stays trivial. Phase 3–4 (before multi-line activation, where scopes become real): land scoped policy + admin UI as its own reviewed workstream with migration. This gets S8's model without churning WS-7 mid-rebuild.

**D6 — Work-item stage integrity.**
S8 stores the full operational stage. Unconstrained, that recreates the current codebase's dual-state-model drift (my Explore finding: two status label maps, two spine models) at product scale: a work item saying "Commercial review" while the referral sits in `evaluation_failed`.
**Resolution: adopt the table, constrain the writes** — derive stages from ground truth where derivable (evidence coverage → Evidence review done; plan approval event → Commercial review done; decision row → Confirmed), store only human-owned facts (owner, waiting reason, escalation, manual stage overrides logged as events), and add a reconciliation invariant test (work-item stage must be reachable from referral status + events). S8's shadow-comparison telemetry already gives the harness for this.

**D7 — Pipeline latency mechanics.**
Same destination, different vehicles: S8 = `after()` best-effort kick (in-process `drainJobs(1)` *or* private worker), F3 = `waitUntil(fetch(cron-url))` with a recent-drain lock. S8's own citation (background promises share the function's timeout) cuts against the in-process variant: a 60s-maxDuration upload route cannot host a 3-minute extraction in `after()`.
**Resolution: converge on S8's *private worker function* variant** — `after(fetch(internal worker route))` where the worker owns a 300s budget — which is functionally my design minus reusing the cron route/secret (cleaner). Keep F3's recent-drain lock (dedupe concurrent kicks), S8's bounded concurrency starting at 2 (their 2 vs my 3 — start lower, config up after observing 429s), and both drafts' insistence that cron + leases remain the durability truth.

**D8 — Confidence display.**
S8: four-part quality panel (evidence coverage, contradictions, unresolved gates, model confidence) and a warning that calibrated bands need validation before use. F3: calibrated bands (High/Likely/Unsure/Low) + reason-code chips + stub honesty. S8's caution is half-right — *un*validated bands presented as probability would be false precision — but their own §6.1 rows surface raw "confidence percentage" demoted to metadata, and a naked 62% is exactly what both our research sets (theirs: CQC AI expectations; mine: Sentinel/reloadux) say fails reviewers.
**Resolution: merge — S8's quality panel is the primary trust surface** (it's made of *verifiable* facts, not model self-report), **F3's bands label the model-confidence component only**, worded as bands-of-model-self-report ("model: unsure"), exact % on hover, explicitly not outcome probability; band thresholds live in config next to `EVAL_CONFIDENCE_THRESHOLD` (one source — F3's P9 threshold-unification carries over). F3's stub-honesty rule (zero-confidence stub never renders a decision badge) survives untouched — S8's draft doesn't cover the stub-Accept trap (`14-referral-accept-testla.png` in my evidence) and the merged audit must keep it.

**D9 — Roadmap duration honesty.**
S8 totals ~20–33 weeks to full multi-line; F3 ~10–13 to cutover with lines after. Mine is optimistic on discovery/sign-off (S8's line slices carry clinical/ops sign-off costs that are real); theirs is heavy pre-proof (P0 1–2wk + P1 3–5wk before any decision-parity work).
**Resolution: F3's Phase-0/1 scope and pace (enablers ≈1wk; inbox + detail read ≈2–3wk) with S8's proof gate (task study + metrics) at the Phase-1 exit; S8's phase structure and durations from decision-parity onward.** Expect ~4–6 months to complex-care cutover; supported living then domiciliary after (adopting S8's order + rationale, still captain-confirmed).

**D10 — What the queue's out-of-scope rows do.** Minor but real: S8 demotes Ofsted via scope/badge; F3 additionally replaces the standard confirm with a terminal "record outside service line / refer on" action so the accept path never renders. Keep F3's stronger version (S8's own decision-safety logic argues for it).

## 3. Gaps BOTH drafts missed

1. **The captain's one-bedroom note reached only one draft.** Muve's units are solo (two-bedroom = staff room, never a second placement), and "some kind of gate in the system we need to remove" is an explicit captain instruction. F3 §8.3.3 handles it (re-scope `capacity`/auto-pass `compatibility_matching` with recorded rationale for solo settings; contract bump catalogued as the later clean fix; captain Q1 asks which gate bites). S8's capacity model is typed and rigorous but line-generic — its supported-living example ("Tenancy/room 6 · household match 2/3 · peer risk") must be re-read through the solo-unit lens: the capacity unit for complex care and (usually) supported living is the **individual one-bed home/tenancy** (schemes may group them), `staff_bedroom` is a feature flag never capacity, and household-compatibility applies only where a scheme is genuinely shared. **Synthesis must carry F3's gate re-scope + captain Q1 into S8's capacity schema.** (Not a fault of S8 — the note arrived mid-task in my session — but it is the single most important asymmetry to fix.)
2. **Email-native intake.** Both drafts mention mailboxes in passing; neither plans the most likely real channel: referrals arriving at a monitored `referrals@` address. WS-2 already parses `.eml` + attachments — a mailbox connector (or even a documented manual .eml drag workflow as v0) is high-value and cheap to specify. Needs a DPIA note (new processor if using an inbound-email service). Propose: named candidate integration, phase 3+, decision owner = captain/DPO.
3. **LA outcome capture (win/loss/withdrawn).** Both drafts track "response prepared/sent" and both *report* win/loss (F3 §5.7, S8 §6.9), but neither specifies where the outcome fact is captured. Needs a small "record LA outcome" action (won / lost-price / lost-capacity / withdrawn / no-response + optional reason) writing an event/`referral_outcomes` row that closes the work item and feeds reporting. Without it the win/loss dashboards in both drafts have no data source.
4. **Concurrent-reviewer presence.** Both drafts have assignment and optimistic version conflicts; neither shows *who else is here now*. A lightweight presence chip ("K.B. viewing · plan edited 2m ago") on the referral header prevents duplicate effort and confusing 409s in a small team. Cheap: heartbeat on the status-cursor poll both drafts already have.
5. **Decision-form draft persistence.** Session expiry or a stale-version bounce mid-rationale loses a reviewer's typed reasoning. Local draft persistence (per referral+user, cleared on confirm) + re-auth returning to state. Neither draft covers it; trivially cheap; disproportionate trust damage when it bites.
6. **In-flight referrals at cutover as an explicit invariant.** S8's shadow comparison covers counts; neither draft states the acceptance criterion outright: *no referral loses stage, owner, SLA clock, or decision state at flag-flip*, verified by a reconciliation report on the cutover PR.
7. **Print/PDF as a first-class output.** S8 has one line (print = light theme); F3 has the evidence pack only. RMs print referral summaries and decision records for handovers, panels, and inspections. The merged plan should commit to print stylesheets for referral workspace + decision record, sharing the evidence-pack renderer.
8. **Accessible rendering of scanned PDFs.** Both drafts ship a document viewer; scanned referrals are image-only and screen-reader-opaque. The extract-as-text *is* the accessible rendering — say so: the evidence matrix doubles as the AT-accessible representation of the source, which makes field-verification coverage an accessibility feature, not just a trust feature.
9. **Document-route protections beyond RBAC.** Both audit access; neither rate-limits or scopes the new source-document endpoint (idempotent UUIDs make enumeration hard, but a per-user rate limit + access-audit sampling costs little on a special-category endpoint). Add cache headers (`private, max-age≈300`) to keep repeated viewing off Blob egress — neither draft budgeted viewer egress.
10. **Degraded-AI-provider UX.** Both improve latency; neither defines what users see when the AI Gateway is down or rate-limited (S8 has a `SystemHealth` component named but undesigned). One honest banner state ("assessment delayed — documents safe, queue will resume") + ops alert threshold. Small, but it's the difference between "broken" and "busy" on a bad day.
11. **Full-text search is explicitly deferred in neither.** Both ship ⌘K by id/LA/initials. "Find the March referral mentioning PEG feeding" needs PII-sensitive content indexing — a real decision (build/defer/never) that should be recorded as deferred-with-rationale rather than silently absent.
12. **Copy governance.** F3 has a jargon kill-list, S8 has per-line terminology adapters and RM sign-off of workflow language; neither names an owner for ongoing copy review as lines multiply. Fold into S8's admin terminology surface with the RM/clinical sign-off step.
